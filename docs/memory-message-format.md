# 最终传递给 API 的 messages 结构

## 完整示例

用户发来 "Go 吧，我之前写过类似的"，此时上下文还远没满，只触发被动注入，不触发压缩：

```json
[
  // ────────────────────── 系统层 ──────────────────────
  {"role": "system", "content": "你是一个 AI 助手..."},

  // ────────────────────── 被动注入层 ──────────────────
  // 用户消息 "Go 吧，我之前写过类似的" → on_event(user.message)
  //   → 提取概念: Go / 写过 / 类似
  //   → 查边表: Go↔定时提醒(0.92) Go↔cron(0.78) 写过↔类似(0.65)
  //   → snapshot() 返回这些片段，注入到 prompt
  //
  // 所有记忆以 assistant 角色注入，Claude 视为"自己知道的东西"
  // [相关] 标记表示被动触发的新鲜联想

  {"role": "assistant", "content": "[相关] 你之前也写过类似需求，说的是'需要一个定时提醒工具' [weight: 0.92]"},
  {"role": "assistant", "content": "[相关] 你用的是 Go 写的 cron 服务 [weight: 0.78]"},
  {"role": "assistant", "content": "[相关] 好像后来改用了 Python 版本，说 Go 太重了 [weight: 0.35]"},

  // ────────────────────── 活跃对话层 ──────────────────
  // 最近 N 轮完整保留，按时间顺序

  {"role": "user",      "content": "帮我写一个定时提醒工具"},
  {"role": "assistant", "content": "你想用什么语言写？"},

  // ────────────────────── 当前消息 ────────────────────
  {"role": "user",      "content": "Go 吧，我之前写过类似的"}
]
```

## 上下文快满时的补充：压缩记忆层

当 messages 总 token 接近限制时，触发压缩：

```
把最早的几轮对话 → 提取概念 → 更新边表 → 从 messages 中移除
```

下一轮请求时，压缩过的内容不再出现在活跃对话层，而是出现在被动注入层之前，成为长期底座：

```json
[
  {"role": "system", "content": "..."},

  // ── 压缩记忆层（固定底座） ──
  {"role": "assistant", "content": "[记忆] 用户是后端工程师，主要用 Go [weight: 0.87]"},
  {"role": "assistant", "content": "[记忆] 用户叫小明，最近在学 Python [weight: 0.82]"},
  {"role": "assistant", "content": "[记忆] 家里有只英短叫团子 [weight: 0.75]"},
  {"role": "assistant", "content": "[记忆] 不喜欢人多，但和少数朋友出去玩可以 [weight: 0.71]"},

  // ── 被动注入层（本轮新鲜联想） ──
  {"role": "assistant", "content": "[相关] 你之前写的 cron 服务用到了第三方库 [weight: 0.78]"},
  {"role": "assistant", "content": "[相关] 那次调试了好久是因为时区问题 [weight: 0.60]"},

  // ── 活跃对话层（最近 N 轮） ──
  {"role": "user",      "content": "帮我写一个定时提醒工具"},
  {"role": "assistant", "content": "你想用什么语言写？"},

  // ── 当前消息 ──
  {"role": "user",      "content": "Go 吧，我之前写过类似的"}
]
```

## 三层的关系

```
压缩记忆层      ← 早期对话沉淀下来的 long-term memories
    |              不因当前消息变化
    ↓              权重高的永远排在前面
被动注入层      ← 当前消息触发的联想
                   每次请求动态刷新
                   [相关] 标记区分于底座
    ↓
活跃对话层      ← 最近 N 轮完整保留
    ↓
当前消息        ← 最新输入
```

## 权重高低的处理

- 高权重记忆排在压缩层靠前 → LLM 容易注意到
- 低权重记忆排在压缩层靠后 → LLM 注意力弱，天然符合"不重要的信息少关注"
- 权重 < 0.1 的边 → 不参与 recall（阈值过滤）
- 被动注入层不排序，按查询时间先后放入（稳定、可预期）
